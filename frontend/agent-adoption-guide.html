Adoption guide

What must change to use this AI agent in another project or with other AI tools

A practical guide for reusing the current Markdown-defined health publishing agent in a different product, backend, domain, or AI provider stack.

Core idea Reuse the workflow

Keep the architecture, execution order, testing, verification, and memory pattern. Replace project-specific contracts and integrations.

Main project changes Domain + data target

Update sources, categories, prompts, templates, database schema, publish API, and environment settings for the new product.

Main AI changes Provider + tool adapter

Swap the model/tool integration, structured output handling, cost limits, and permissions while preserving the workflow contract.

Repo anchors Markdown-defined system

The current repo is driven by `AGENT.md`, `docs/workflows`, `docs/prompts`, `docs/templates`, and `config/*.md`.

Reuse

What can stay the same

  • Layered structure: keep the architecture, runner, planner, execution, tester, verifier, and publisher separation described in `README.md` and `AGENT.md`.
  • Markdown-defined contracts: keep the pattern where behavior is driven by Markdown files under `docs/`, `config/`, and `logs/`.
  • Execution sequence: source fetch → filtering → duplicate check → research → writing → testing → verification → publish → memory/logging.
  • Operational principles: use memory reuse, structured outputs, validation gates, retry/skip behavior, and detailed run logs.

Project adaptation

Changes required for another project

Mission and audience

Replace the healthcare-professional mission with the target product audience, business goals, and domain restrictions.

Source strategy

Replace health bodies and journals with the trusted sources that matter for the target project.

Planner rules

Change topic selection, scoring, categories, and publish-worthiness rules in planner and verifier documents.

Template and tone

Change the article/report structure, metadata fields, formatting rules, and content voice.

Integration

Data model and publishing changes

  • Publish destination: the current repo publishes to MySQL targets like `mydrscripts_new.blog_category` and `mydrscripts_new.blog_master`. Another project may require a different schema, a CMS API, a queue, or object storage.
  • Operational database: create an equivalent to `health_ai_agent` for memory reuse, run logs, retries, and execution visibility.
  • Environment variables: update `.env` expectations for host, port, credentials, publish DB name, agent DB name, and any storage keys.
  • Publishing scripts: replace assumptions inside workflow/publish scripts so insert/update payloads match the new project fields.

Contracts

Prompt, role, and template changes

The current agent depends on prompt and role files such as `docs/prompts/FETCH_NEWS_PROMPT.md`, `FILTER_PROMPT.md`, `RESEARCH_PROMPT.md`, `WRITE_PROMPT.md`, `VALIDATE_PROMPT.md`, plus specialist roles under `docs/roles/`.

  • Rewrite each prompt to match the new domain vocabulary and output expectations.
  • Replace healthcare evidence language with the quality rubric that fits the new product.
  • Update category mapping, duplicate rules, approval gates, and required output schema.
  • Update `docs/templates/BLOG_TEMPLATE.md` or replace it with the target format entirely.

Operations

Scheduler, limits, and observability changes

The current schedule is defined in `config/SCHEDULER.md` with a recurring run, run limits, and timeout expectations.

If the target project is started through an agent interface, preserve a clear trigger contract as well: keep one primary command such as start-agent, document equivalent phrases like start, start agent, start work, and start-work, and decide whether legacy prompts such as Read AGENT.md or Start AGENT.md should also remain valid.

Frequency

Adjust the cron cadence to match how often the new domain changes.

Run limits

Change max topics, max outputs, and timeout rules for the new workload size.

Visibility

Preserve run logs, status events, and dashboard exposure so failures stay observable.

AI tooling

Changes required for other AI tools or model providers

  • Provider adapter: replace the current model/tool calling layer with the new provider SDK, API shape, auth, retries, and timeout handling.
  • Structured output compatibility: confirm the new model can reliably produce the schemas expected by the tester and verifier stages.
  • Tool permissions: ensure the new runtime supports web access, local command execution, file access, and any browser-assisted verification you require.
  • Cost and latency policy: update token/cost limits, batching behavior, and fallback rules based on the new provider.
  • Multimodal or voice features: if the target product includes voice or note generation, map equivalent services to patterns like the backend `OpenAIService.js`, `AiTalkService.js`, and `MedicalScribeService.js` described in architecture docs.

Takeaway

Best way to reuse this agent

The safest path is to treat this repository as a reusable agent framework: keep the layered workflow, memory/logging discipline, and publish gate, but replace the domain contracts, integrations, data model, and AI provider assumptions that are specific to MyDrScripts health publishing.
